# Writing Patterns / 财经科技公众号写作模式

This file distills reusable writing patterns from strong Chinese finance, technology, venture, and industry writing. Use it before rewriting the article.

## 一、样本观察

这些模式来自对财经科技内容的公开样本观察,包括 36氪、虎嗅、晚点 LatePost、远川研究所、财经十一人、第一财经 YiMagazine 等类型的内容。目标不是复刻某一家媒体的语气,而是提炼可迁移的写作动作。

共同点:
- 开头少讲背景,先给读者一个具体场景、异常动作、反常数字或明确冲突。
- 文章不急着下大判断,先搭事实阶梯:谁在做、发生了什么、为什么现在变了。
- 小标题承担推进功能,不是目录功能。
- 每一节都有一个可复述的判断,但判断贴着事实走。
- 引语用于还原人和现场,不是替作者喊口号。
- 数据、图表、案例、人物动作各司其职。
- 收束落在行业位置、组织选择或商业代价上,不落在空泛想象空间。

## 二、六种开场模式

### 1. 异常动作开场

用一个反常动作切入,让读者先看到变化。

模板:
```
过去三个月,XX 团队做了一件过去不会做的事:____。
```

适用:
- 公司转向
- 新产品上线
- 组织变化
- 商业化变化

写法要求:
- 第一段必须有主体和动作。
- 第二段解释为什么这个动作反常。
- 第三段才扩大到行业背景。

### 2. 小场景开场

从一个具体工作场景进入,适合技术、产品、ToB、人物稿。

模板:
```
周二晚上 10 点,____ 还在看一张 ____ 表。
```

写法要求:
- 场景要能引出文章主问题。
- 不写虚构情绪,只写可观察动作。
- 场景后立刻收束成问题。

### 3. 反常数字开场

用一个数字制造阅读入口,但数字必须来自草稿或公开来源。

模板:
```
____ 增长了,但 ____ 没有同步增长。
```

写法要求:
- 数字进入正文前必须进数字清单。
- 敏感或竞品数字不进标题、封面、摘要和图。
- 数字后必须解释口径,避免截图脱离上下文。

### 4. 时间节点开场

用一个时间点作为变化的切口。

模板:
```
转折发生在 ____。
```

写法要求:
- 时间点后要跟具体事件,不能只写时代变化。
- 不用"风口""浪潮"作为解释。

### 5. 人物动作开场

适合创始人、投资人、技术负责人。

模板:
```
____ 最近把每周例会的第一项,改成了 ____。
```

写法要求:
- 人物动作要指向组织选择。
- 引语后置,不要开头就放大段引语。

### 6. 旧认知失效开场

适合行业拆解和复盘。

模板:
```
过去,行业默认 ____。现在,这个判断开始失效。
```

写法要求:
- 先解释旧认知为什么曾经成立。
- 再解释什么变化让它失效。
- 不使用二元转折公式。

## 三、信息阶梯

强稿件通常不是平铺材料,而是把信息按读者理解顺序排列。

推荐顺序:
1. 入口:场景、动作、数字或时间点。
2. 异常:它和过去有什么不同。
3. 主角:谁在变化,谁被迫变化。
4. 机制:变化是怎么发生的。
5. 证据:客户、收入、组织、产品、数据或案例。
6. 代价:这个选择牺牲了什么。
7. 外推:它对行业意味着什么。
8. 收束:读者带走的判断。

每一节只推进一层。不要在同一节里同时写背景、机制、竞品和愿景。

## 四、小标题模式

小标题要像路标,读者扫一眼知道文章正在往哪里走。

可用:
- "订单先变,岗位才变"
- "搜索之后,工作还没结束"
- "客户页要先回答三个问题"
- "真正贵的,是切换成本"
- "大模型变便宜后,组织问题变贵了"

少用:
- "行业背景"
- "产品优势"
- "技术能力"
- "商业价值"
- "未来展望"

## 五、段落推进

### 标准推进段

```
事实句。
解释句。
判断句。
```

示例:
```
客户不再只问设备价格,他们开始把现场视频发给供应商。
这意味着供应商必须解释异常为什么发生,交付责任也从硬件延伸到结果。
硬件仍是入口,服务开始决定复购。
```

### 场景解释段

```
具体动作。
背后的工作流。
读者能理解的代价。
```

示例:
```
销售打开客户页时,先看到三条风险线索。
每条线索后面都挂着原始会议纪要和邮件片段。
这样他不用在十几个系统里重新拼一遍客户背景。
```

### 判断收束段

```
事实范围。
判断。
边界。
```

示例:
```
这类产品解决的是高频但低耐心的工作。
它的价值不在多一个页面,而在少一次切换。
如果数据源不完整,页面再漂亮也只能变成新的待办清单。
```

## 六、引语模式

好引语提供现场感、判断或冲突。它不负责解释全文。

可用引语:
- "客户不关心模型名字,只关心凌晨两点会不会误停线。"
- "我们不是想让销售多看一个系统,而是让他少问三个人。" 这类句式如果保留,必须来自真实受访者原话;代拟时改写。
- "现在的问题不是有没有数据,而是谁愿意为数据负责。" 真实原话可保留;代拟时改成平实句。

代拟改写:
- "销售少问三个人,这个页面才有价值。"
- "数据有人负责,系统才可能被每天打开。"

规则:
- 代拟引语必须在发布说明中标注。
- 全文 1–3 个引语块。
- 引语前后要有正文承接,不要把引语孤零零放在节末。

## 七、四种文章骨架

### A. 公司报道型

适合融资、公司故事、产品发布、组织调整。

结构:
标题 → 异常动作 → 为什么反常 → 公司选择 → 机制/产品 → 证据 → 行业位置 → 关于栏。

写作重点:
- 先写变化,再写背景。
- 主角选择要具体。
- 证据不要只写"获得认可",要写客户、部署、收入、组织或产品动作。

### B. 行业拆解型

适合赛道、产业链、商业模式。

结构:
旧认知 → 新变化 → 关键变量 → 受益者/受损者 → 公司案例 → 结论。

写作重点:
- 先拆变量,再讲观点。
- 不把所有公司写成同一种命运。
- 结尾给边界条件。

### C. 产品机制型

适合 SaaS、AI 应用、技术产品、解决方案。

结构:
具体工作痛点 → 旧流程 → 新机制 → 信息图 → 使用场景 → 限制 → 下一步。

写作重点:
- 每个技术词后面跟一个人能理解的动作。
- 图放在机制段后。
- 产品价值落在时间、成本、错误率、切换成本或责任归属上。

### D. 人物访谈型

适合创始人、投资人、技术负责人。

结构:
人物动作 → 关键选择 → 过去经历 → 当下判断 → 争议/代价 → 引语收束。

写作重点:
- 人物不是观点机器,要有动作、选择和代价。
- 不写履历流水账。
- 引语要有人味,但不能替代事实。

## 八、AI 腔替换表

| 弱写法 | 改写方向 |
|---|---|
| 这不仅是技术升级,更是商业模式重塑 | 写清升级了哪个环节,影响哪项成本或收入 |
| 形成完整闭环 | 写清哪个反馈回到哪个流程 |
| 赋能行业客户 | 写清客户少做了哪一步,或多看见了什么 |
| 打开想象空间 | 写清下一步可能卖给谁,以及难点是什么 |
| 生态协同 | 写清谁和谁一起做了什么 |
| 深度融合 | 写清两个系统交换了哪些数据 |
